## Notes for Claude sessions on this repo
- **All four live SQLite databases with real data are plain files in the project root**, not under `prisma/`: `interlex.db`, `auth.db`, `library.db`, `corpus.db` (confirmed via `.env`'s `DATA_DATABASE_URL`/`AUTH_DATABASE_URL`/etc. and `.env.development`'s `SQLITE_DB`). `prisma/*.schema.prisma` are schema *source* files only — the actual `.db` files they point at live at the repo root, not inside `prisma/`. Before writing any DB-touching script or migration, `cat .env` to confirm the current path rather than assuming. Treat these files with the same care as production data — back up before any raw-SQL DDL/migration (see `interlex.db.backup-before-indexes` for the pattern used during the Phase 2 index work).
- [ARCHITECTURE.md](ARCHITECTURE.md) has an up-to-date system diagram plus a "Known Issues & Technical Debt" section — check it before assuming something is broken or missing; several items (word frequency/CEFR, corpus DB, four-database split) are already shipped despite older docs/instincts suggesting otherwise.
- 2026-07-22: ran a security/architecture audit and fixed Phase 1 (SQL injection, missing permission check, two unauthenticated endpoints, non-constant-time HMAC compare), then Phase 2 items 6-7 (missing indexes, FTS5 trigram search) and item 5 (bidirectional relations, via new `lib/relations.ts` — `fetchSymmetricRelations`/`saveSymmetricRelation`, used across all 7 read/write sites for the 11 relation tables).
- 2026-07-24: fixed the Phase 3 grammar-engine endings bug. The live `ending_allophones` DB table had already been manually corrected via `/admin/endings` over time (the DB-override mechanism in `endingLoader.ts` is real, not a decoy) — the bug was that the *hardcoded* registries (`endingsRegistry.ts`, `adjective/index.ts`) and `seed-endings.ts` were stale copies of the original Proto-Slavic values. Extracted the DB's corrected values back into the registries, fixed the seed script (now idempotent/safe to re-run), migrated the nasal vowel `ǫ`→`ų` across every modern-ISV code path (confirmed with maintainer — leave `lib/proto.ts`'s genuine Proto-Slavic input alone), and deleted two fully-dead duplicate registries found along the way (`noun/index.ts`'s unused `SLAVIC_ENDINGS_REGISTRY`, the whole `adjective/adjective.ts` file). See AGENTS.md's "RESOLVED: Grammar Engine..." section for the full writeup, including what's still open (verb/pronoun endings not independently verified, four-tones accent engine has a separate unrelated bug, no test runner yet).
- **`lib/grammar/verb/index.ts` and `lib/grammar/verb/conjugator2.ts` are two independently-live verb conjugators**, not a dead duplicate pair like noun/adjective turned out to be — the engine (`processVerb`) uses `verb/index.ts`, but `app/words/[id]/Word.tsx` imports `conjugateFullVerb` directly from `verb/conjugator2.ts`. Any future verb-ending fix needs to touch both, and don't assume a second copy of something is automatically dead — check importers first (this project has both patterns).
- Remaining from Phase 3/4: verb present/aorist/imperfect/imperative and `numeral_two`/`collective_*` endings were checked against DB and found to already look correct (no jers) but not linguist-verified; `morphology.test.ts`/`declineNoun.test.ts` still have no real runner (vitest not set up); N+1 queries on `/words/[id]`; widespread `any`. Tracked in ARCHITECTURE.md's "Known Issues" — don't re-discover these from scratch, pick up where that list left off.
- Any new code touching the synonyms/antonyms/hypernyms/.../conclusions tables must go through `lib/relations.ts`, not a fresh `WHERE sourceId = ?` query — that pattern is exactly what caused the bidirectionality bug.
- When adding any API route under `app/api/**` that mutates lexical/relation data, always gate it with `checkPermission(session, Feature.X)` for the specific action (see `lib/permissions.ts`, `config/features.ts`) — a session check alone is not enough, and this exact gap is what caused the word-relations vulnerability above.
- **`prisma migrate dev` is currently unsafe on the `data` schema** — it detects drift (`morpheme_allophones`, `proto_slavic_words` changed outside any migration) and offers to reset the DB, which would delete all lexical data. Don't run it until this is reconciled. The index migration above was applied as raw `CREATE INDEX IF NOT EXISTS` SQL directly against `interlex.db` instead — follow that pattern for further schema-only (no new Prisma-modeled fields) changes until the drift is fixed.
- A pre-migration backup lives at `interlex.db.backup-before-indexes` (repo root, gitignored via `*.db`... actually NOT matched by that pattern since the suffix isn't `.db` — it's untracked but not ignored, so don't `git add -A` near it). Safe to delete once the index/FTS5 changes are confirmed stable in production.
- **The system `sqlite3` CLI on a host and the SQLite bundled inside `better-sqlite3` (what the app actually runs on) can be different builds with different capabilities.** Learned this the hard way: production's `sqlite3` CLI lacked the FTS5 trigram tokenizer and failed mid-migration when applying `2026-07-22-add-indexes-and-fts5-trigram.sql` there, dropping `lexemes_text`/`lexeme_allophones_text` without being able to recreate them (search broke until `2026-07-22-emergency-restore-fts5.sql` was run). For any DB DDL that depends on an optional SQLite feature (FTS5 tokenizers, JSON1, etc.), prefer a Node script using the project's own `better-sqlite3` (`npx tsx scripts/db/....ts`, see `2026-07-22-add-indexes-and-fts5-trigram.ts` for the pattern: probe for the feature in a throwaway `temp` table before touching real tables) over a raw `.sql` file run through the host's system CLI.
